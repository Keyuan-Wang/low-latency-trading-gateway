# Phase 6 Benchmark Handle Migration

## Context

`phase6_engine_handle_refactor_plan.md` moved the matching core off the `absl::flat_hash_map<order_id, Order*>` cancel index and onto engine-issued `OrderHandle`s. `cancel_order()` and `modify_order()` now take a handle, and `add_limit_order()` returns the handle of any resting remainder in `AddResult`.

That refactor invalidated the benchmark harness, which previously tracked live orders by business `order_id` and resolved cancel/modify targets through a hash map. This report records how `bench_hft_macro` and `bench_overall` were migrated to the handle contract, the design problem that migration exposed, the approach chosen to solve it, and the validation results.

## The Problem The Migration Exposed

The plan's "Benchmark Implications" section flagged the core difficulty: a handle obtained from a dry-run/planning book is not automatically valid in the measured book, because each `OrderBook` owns its own `OrderPool`. The plan offered two options:

1. Make slot allocation deterministic so the planning and measured books assign identical handles for the same replayed operation stream.
2. Track two handles per logical order, one per book.

A third, tempting but wrong option also had to be ruled out: keep tracking orders by `order_id` and perform an `order_id -> handle` lookup inside the timed `RunOp()` window. That would relocate the very hash lookup the refactor removed from the engine straight back into the benchmark's measured path, producing a false optimization where the headline number improves only because the cost moved out of frame.

## Approach Chosen: Deterministic Dual-Build (Option 1)

The migration takes option 1. The timed window performs pure `OrderBook` operations with handles resolved entirely during `Setup()`:

```cpp
case PendingOp::kCancel: {
    auto const code = book_->cancel_order(op.target_handle);
    if (code == llmes::matching_core::ErrorCode::Success) {
        ++ok;
        outcome.bucket = OpBucket::kCancelHit;
    } else {
        outcome.bucket = OpBucket::kCancelMiss;
    }
    break;
}
```

There is no `book_handles_.find(...)` anywhere in `execute_pending()`. Every cancel and modify carries a precomputed `op.target_handle`.

### How handles stay valid across two books

`bench_hft_macro::Setup()` runs the workload twice on a fresh pool:

1. Warmup populates `resting_orders_` tracking, then `book_` is built from that tracking via `build_book_from_tracking()`.
2. `resting_orders_` is snapshotted into `base_resting_orders`.
3. The measured batch is pre-generated by executing each candidate operation on `book_` itself. Adds, cancels, modifies, and market orders mutate `book_`, and the real `OrderHandle` returned for each resting order is captured into the corresponding `PendingOp`.
4. After pre-generation, `resting_orders_` is restored from `base_resting_orders` and `book_` is rebuilt from that identical snapshot.
5. The timed run replays `pending_` against the rebuilt `book_`.

The handles captured in step 3 remain valid in step 5 because `OrderPool` is deterministic: for an identical sequence of `acquire()` / `release()` calls on a fresh pool, it returns the identical sequence of slot indices. The two builds start from the same sorted resting set and replay the same operation order, so the same logical order receives the same handle in both passes.

`bench_overall` reaches the same guarantee with the explicit two-book form: it keeps a `planning_book_` alongside `book_`, both prefilled identically, and uses the planning book during `Setup()` to discover handles and validate target selection. Because both books are built deterministically from the same parameters, the planning handle stored in each `PendingOp` is valid on the measured book.

### Ordering detail: cluster cancels

Spontaneous cancel clusters are generated inline while a primary cancel is being produced, so they are applied to the dry-run book immediately but must be appended to `pending_` after the primary op. `pregen_queue_` is drained with `pop_back()`, so the cluster ops are reverse-appended to preserve the exact order in which they were applied to the dry-run book:

```cpp
// pregen_queue_ is drained with pop_back(); reverse append preserves
// the same cancel order already applied to the dry-run book.
pregen_queue_.insert(pregen_queue_.end(),
                     cluster_ops.rbegin(), cluster_ops.rend());
```

If the replay order diverged from the dry-run apply order, the pool's acquire/release interleaving would differ and the precomputed handles would desync.

### Other migration changes

- Operation bucket classification (`add_rests`, market `filled_quantity`, market `levels_touched`) is precomputed during `Setup()` and stored on the `PendingOp`, so the timed window does not recompute it. The dead `apply_book_trade_fills()` helper and the obsolete `planning_handles_` map were removed.
- The modify path no longer has a `kModifyMiss` outcome: all pre-generated modifies are validated hits, so the timed window always issues a successful `modify_order(handle, ...)`.
- Market sizing now guards `total * 0.8` with `std::max<std::uint64_t>(1, ...)` to avoid a zero-quantity market order at low depth.

## The Implicit Invariant And Its Fragility

Correctness rests on one invariant that is currently enforced only by construction, not by an assertion:

> Every book mutation performed during pre-generation is captured in `pending_` exactly once, in the same relative order.

If a future edit introduces a pre-generation mutation that is not mirrored 1:1 into `pending_`, or reorders the replay relative to the dry run, the pool's allocation sequence diverges and the precomputed handles silently point at the wrong slots. Under the strict benchmark assumption (no generational check on the slot-index-only prototype), that desync would not necessarily crash; it could quietly produce meaningless timings.

Recommended safety belt: count cancel/modify failures during the timed window and assert the count is zero at teardown. A precomputed handle that fails to resolve is a direct signal that the dry-run/replay sequences have diverged, and it surfaces the bug immediately instead of letting the benchmark report a confidently wrong number.

## Validation

- ASan build of `bench_hft_macro` and `bench_overall`: clean compile.
- ASan run, `--orders 2000 --levels 50 --batch-size 5000 --warmup-iters 1 --iters 2`: zero memory errors on both binaries, with `ok == batch_size * (warmup_iters + iters)`. Every cancel and modify resolved its precomputed handle and succeeded, which is the direct correctness signal for the dual-build handle replay; any desync would drop `ok` or trip ASan.
- Core gtest suite: 8/8 passing.

## Relationship To The Engine Refactor

This migration is the benchmark-side completion of the plan's items 7 and 8 ("Update `bench_hft_macro` to track engine handles", "Re-run correctness tests and smoke benchmarks"). It also realizes the plan's "Benchmark Scope" rule: handle tracking and target selection happen in `Setup()`, and the timed window models only the matching core's real input on the cancel/modify paths, which is an already-resolved handle. The gateway-side `client_order_id -> handle` lookup remains explicitly out of scope for this matching-core benchmark.
